## 1. 审查机制概述

### 1.1 为什么要审查机制

| 问题 | 没有审查 | 有审查 |
|------|---------|--------|
| 文档不一致 | 各文档相互矛盾 | 上层文档是准则，下层必须符合 |
| 文档不完整 | 缺内容、缺字段 | 检查清单确保完整 |
| 逻辑错误 | 实现时才发现设计问题 | 审查时发现并修正 |
| 重复内容 | 多处定义同一概念 | 唯一来源，避免矛盾 |

### 1.2 审查原则

| 原则 | 说明 |
|------|------|
| **下层符合上层** | L2 文档必须符合 L1，L1 必须符合 L0 |
| **唯一来源** | 同一概念只在一处定义，其他地方引用 |
| **清晰边界** | 每个文档有自己的职责范围 |
| **可追溯** | 每个决策有据可查 |

## 2. 文档层级

### 2.1 层级定义

```
┌─────────────────────────────────────────────────────────┐
│  L0：宪法（最高）                                        │
│                                                         │
│  SPEC.md                                                │
│  总架构文档，框架定义，所有其他文档必须符合                │
│                                                         │
│  定义：                                                  │
│  ├─ 三层架构（UI / Core / Modules）                    │
│  ├─ 模块间通信通过 Core，不直接通信                      │
│  ├─ 开源协议（Core=Apache, 基础模块=GPL, 高级=商业）    │
│  └─ 模块必须实现 Module trait                           │
└─────────────────────────────────────────────────────────┘
                          │
                          │ L1 必须符合 L0
                          ▼
┌─────────────────────────────────────────────────────────┐
│  L1：法律                                                │
│                                                         │
│  SPEC-Core.md          → Core 核心规格                   │
│  SPEC-Core-API.md      → WebSocket API 规格             │
│  SPEC-Plugin-Interface.md → 模块插件接口规格             │
│                                                         │
│  这些文档定义 Core 和模块如何交互                         │
│ L1 文档之间不能相互矛盾                                  │
└─────────────────────────────────────────────────────────┘
                          │
                          │ L2 必须符合 L1
                          ▼
┌─────────────────────────────────────────────────────────┐
│  L2：法规                                                │
│                                                         │
│  MOD-p2p.md                                            │
│  MOD-tor.md                                            │
│  MOD-i2p.md                                            │
│  MOD-crdt.md                                           │
│  MOD-crypto.md                                         │
│                                                         │
│  具体模块实现规格，必须符合 L1 的接口定义                 │
└─────────────────────────────────────────────────────────┘
                          │
                          │ L3 必须符合 L0/L1
                          ▼
┌─────────────────────────────────────────────────────────┐
│  L3：地方法规                                            │
│                                                         │
│  SPEC-Install-Flow.md   → 安装引导流程                   │
│  SPEC-UI-Android.md    → Android UI 规格                │
│  SPEC-UI-iOS.md        → iOS UI 规格                   │
│  SPEC-UI-Desktop.md     → 桌面端 UI 规格                 │
│                                                         │
│  平台特定实现，必须符合 L0/L1 的框架定义                 │
└─────────────────────────────────────────────────────────┘
```

### 2.2 层级规则

| 规则 | 说明 |
|------|------|
| **上级优先** | 下层文档与上层冲突时，以上层为准 |
| **不得越级** | L2 不能定义 L1 的内容，L1 不能定义 L0 的内容 |
| **引用而非复制** | 下层引用上层的定义，不重复定义 |
| **范围明确** | 每个文档有自己的职责边界 |

## 3. 审查流程

### 3.1 标准审查流程

```
┌─────────────────────────────────────────────────────────┐
│                    标准审查流程                           │
│                                                          │
│  1. 编写 ──▶ 2. 自审 ──▶ 3. 交叉审查 ──▶ 4. 批准发布    │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

#### 第一步：编写

```
作者编写文档
    │
    ├─ 确定文档属于哪个层级（L0/L1/L2/L3）
    │
    ├─ 对照该层级的检查清单（见附录）
    │
    └─ 完成初稿
```

#### 第二步：自审

```
作者对照检查清单自我审查
    │
    ├─ 检查清单逐项打勾
    │
    ├─ 标记任何不符合项
    │
    ├─ 如果有不符合项 → 修正后再提交
    │
    └─ 无不符合项 → 进入第三步
```

#### 第三步：交叉审查

```
由同层级其他文档的作者或架构师审查
    │
    ├─ L2 文档 → 由 L1 作者审查
    │
    ├─ L1 文档 → 由架构师或其他 L1 作者审查
    │
    ├─ L0 文档 → 由核心团队共同审查
    │
    └─ 审查结果：
         ├─ 通过 → 进入第四步
         ├─ 需修改 → 返回作者修改
         └─ 有重大问题 → 升级处理
```

#### 第四步：批准发布

```
审查通过后，由负责人批准发布
    │
    ├─ L0 → 核心团队批准
    │
    ├─ L1 → 架构师批准
    │
    ├─ L2 → L1 作者批准
    │
    └─ L3 → L1 作者或项目经理批准
```

### 3.2 快速审查流程

适用于小型修改（如修复错别字、补充说明）：

```
修改 → 自审 → 交叉确认 → 发布
             ↑            │
             └────────────┘
```

### 3.3 紧急审查流程

适用于紧急修复：

```
紧急问题 → 快速审查 → 临时批准 → 后续补审
                              ↓
                        标记为"临时"
                        48小时内补审
```

## 4. 审查内容

### 4.1 L0（宪法）审查内容

| 检查项 | 说明 |
|--------|------|
| **框架完整性** | 三层架构定义清晰 |
| **模块分类** | 内置模块 vs 外部模块 定义明确 |
| **通信原则** | 模块间通过 Core 通信，不直接通信 |
| **开源协议** | 各组件许可证定义明确 |
| **向下引用** | 引用 L1 文档，不定义 L1 细节 |
| **无矛盾** | 自身内部无逻辑矛盾 |
| **可执行** | 框架定义足够具体，可指导 L1 编写 |

### 4.2 L1（法律）审查内容

| 检查项 | 说明 |
|--------|------|
| **符合 L0** | 符合 SPEC.md 的框架定义 |
| **边界清晰** | 本文档职责范围明确，不越权 |
| **无横向冲突** | 与其他 L1 文档无矛盾 |
| **完整定义** | 本层级应定义的内容完整 |
| **引用 L0** | 引用 L0 定义，不复制 L0 内容 |
| **供 L2 引用** | L2 需要的内容已明确定义 |
| **可实现** | 技术方案可实现 |

### 4.3 L2（法规）审查内容

| 检查项 | 说明 |
|--------|------|
| **符合 L1** | 符合对应 L1 文档的接口定义 |
| **Module trait** | 必须实现的方法完整 |
| **manifest.json** | 清单字段完整 |
| **JSON Schema** | 方法参数和返回值 Schema 完整 |
| **事件定义** | 推送的事件类型完整 |
| **适用范围** | 适用/不适用场景已列出 |
| **无横向冲突** | 与其他 L2 文档无矛盾 |
| **依赖声明** | 依赖的 L1 接口已明确 |
| **审查清单** | 文档末尾的审查清单已填写 |

### 4.4 L3（地方法规）审查内容

| 检查项 | 说明 |
|--------|------|
| **符合 L0** | 符合 SPEC.md 框架 |
| **符合 L1** | 符合相关 L1 文档定义 |
| **平台适配** | 平台特定内容完整 |
| **用户流程** | 用户操作流程清晰 |
| **错误处理** | 异常情况有处理方案 |
| **界面设计** | 界面布局合理 |
| **可实现** | 方案可实现 |

### 5.2 问题追踪

| 问题类型 | 处理方式 |
|---------|---------|
| **不符合上层** | 必须修正，否则不能发布 |
| **不完整** | 必须补充，否则不能发布 |
| **逻辑错误** | 必须修正，否则不能发布 |
| **建议** | 可以先发布，后续迭代修正 |

## 6. 文档状态

### 6.1 状态定义

| 状态 | 说明 | 可否发布 |
|------|------|---------|
| `草稿` | 正在编写中 | ❌ |
| `待审` | 自审通过，等待交叉审查 | ❌ |
| `审查中` | 正在被审查 | ❌ |
| `需修改` | 审查发现问题，需修改 | ❌ |
| `已批准` | 审查通过，等待发布 | ❌ |
| `已发布` | 正式发布 | ✅ |
| `已废弃` | 已过时，不再使用 | ❌ |
| `临时` | 紧急审查通过，临时使用 | ⚠️ |

### 6.2 状态流转

```
草稿 ──▶ 待审 ──▶ 审查中 ──▶ 已批准 ──▶ 已发布
              │         │
              │         ▼
              │      需修改
              │         │
              └─────────┘
              
紧急：草稿 ──▶ 临时 ──▶ 已发布
                    │
                    ▼
              48小时内补审
```

## 7. 审查责任

### 7.1 角色定义

| 角色 | 职责 |
|------|------|
| **作者** | 编写文档，完成自审 |
| **审查人** | 执行交叉审查，提出问题 |
| **批准人** | 最终批准文档发布 |
| **架构师** | 把握 L0 正确性，指导 L1 |

### 7.2 审查权限

| 文档层级 | 作者 | 审查人 | 批准人 |
|---------|------|--------|--------|
| L0 | 核心团队 | 核心团队共同 | 核心团队一致 |
| L1 | 架构师/开发者 | 架构师 | 架构师 |
| L2 | 开发者 | 对应 L1 作者 | L1 作者 |
| L3 | UI 设计师 | L1 作者/PM | L1 作者/PM |
