## 1. 项目愿景

Cloudfin 是一个**模块化跨平台通信与同步框架**。

类比目标：

- **Redis** —— 纯 API/CLI 控制，程序化驱动
- **无头浏览器** —— 自动化脚本接管一切

Cloudfin 不做封闭生态。**核心目标：让任意多个设备在任意网络环境下安全、低延迟地同步数据。**

UI 与 Core/Modules 严格分离：UI 是独立进程，可以是任意平台的原生应用，也可以是 CLI 工具。

一切围绕这个目标展开：数据怎么存、怎么传、怎么一致、怎么扩展，都服务于这个核心承诺。

## 2. 核心功能需求

### 2.1 跨平台守护进程

- **全平台运行：** Android / iOS / Windows / macOS / Linux
- **无原生 UI：** Core 本身不包含任何 UI，纯后台进程
- **常驻自启：** 开机启动，进程守护，断线重连

### 2.2 模块化插件系统

- 每个功能封装为独立模块（`.so` / `.dylib` / `.dll`）
- 模块可按需安装、加载、卸载，无需重启 Core
- 模块与 Core **严格解耦**：模块崩溃不拖垮 Core，Core 重启不影响模块加载策略
- 模块通过 Core 暴露的统一接口访问能力

### 2.3 UI 与 Core 分离

- UI 是**独立进程**，与 Core 完全解耦
- UI 与 Core 之间通过**单一 WebSocket 连接**通信
- UI 可以是任意平台的原生应用：Android App、iOS App、Qt、Electron、控制台 CLI……形态不限
- 一套协议，多端复用，开发者可以自由选择 UI 技术栈

### 2.4 P2P 组网能力

- **去中心化节点发现：** 无需中心服务器，节点自主组网
- **NAT 穿透：** 支持中继（relay）与打洞（hole punching），覆盖全网络场景
- **端到端加密：** 所有 P2P 流量默认加密，防止中间人

### 2.5 CRDT 文档同步

- **最终一致性：** 无论先到后到，最终收敛到同一状态
- **离线可用：** 网络中断不影响本地读写，重连后自动合并
- **冲突自动合并：** 无需人工干预，无主节点

## 3. 基本技术路线

### 3.1 为什么用 Rust

| 理由 | 说明 |
|------|------|
| 性能优先 | 与 C/C++ 同级，适合网络 I/O 密集场景 |
| 内存安全 | 无 GC，无数据竞争，无 use-after-free |
| 跨平台 FFI | 与模块系统天然契合 |
| 现代包管理 | Cargo 一站式依赖与构建 |

Rust 是少数能够在**高性能**与**内存安全**之间不妥协的语言，对于需要长期运行、跨平台分发的守护进程，这是最优解。

### 3.2 为什么五层分离

```
┌─────────────────────────────────────────────────────┐
│  第一层：UI 层（平台原生，用户体验优先）                  │
│  Android App / iOS App / Qt / CLI …                  │
└─────────────────────────┬───────────────────────────┘
                          │ WebSocket（单一协议）
┌─────────────────────────▼───────────────────────────┐
│  第二层：Core 层（跨平台，纯逻辑，无 UI）                │
│  连接管理 / 路由 / 模块调度 / 配置                       │
└─────────────────────────┬───────────────────────────┘
                          │ 统一接口（Core API）
┌─────────────────────────▼───────────────────────────┐
│  第三层：通信层 Modules（网络能力，按协议分层）             │
│  p2p（点对点）/ tor（洋葱路由）/ i2p（隧道）              │
└─────────────────────────────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────┐
│  第四层：加密层 Modules（安全能力）                        │
│  crypto（端到端加密管道）                                │
└─────────────────────────────────────────────────────┘
                          │
┌─────────────────────────▼───────────────────────────┐
│  第五层：同步层 Modules（数据能力）                        │
│  crdt（文档协同）/ storage（持久化存储）                   │
└─────────────────────────────────────────────────────┘
```

- **UI 层：** 平台相关，用户体验优先，换技术栈不影响 Core
- **Core 层：** 跨平台通用，专注连接、路由、安全、模块调度
- **通信层 Modules：** 网络能力，按协议类型分为 p2p/tor/i2p
- **加密层 Modules：** 安全能力，提供端到端加密管道
- **同步层 Modules：** 数据能力，提供文档协同与持久化存储

五层各司其职，任一层换语言、换框架，其他层**无需修改**。

### 3.3 为什么纯 WebSocket 通信

- **全平台通用：** Android / iOS / Desktop / Web，一个协议打天下
- **双向实时推送：** Server → Client / Client → Server 同时支持，无需轮询
- **HTTP/1.1 兼容：** 天然穿透企业代理，兼容现有网络环境

WebSocket 是 UI 与 Core 之间通信的**最小公分母**。

### 3.4 为什么模块间不能直连

模块间**禁止直连**，所有通信必须经过 Core 中介。理由：

1. **安全隔离：** 模块权限由 Core 显式授予，无 Core 授权则无法跨模块访问数据
2. **可信中介：** Core 是唯一可信实体，防止恶意模块伪装成合法模块
3. **防止隐性依赖：** 若模块间直连，会形成未声明的耦合，任意模块升级都可能破坏其他模块

### 3.5 为什么选择 Open Core 模式

- **基础功能开源（AGPL-3.0）：** p2p、crdt、storage 等核心模块开源，社区验证安全性与稳定性
- **高级功能商业授权：** 加密管道、I2P 支持等模块商业授权，为持续投入提供资金来源
- **避免协议锁死：** 核心协议栈开源，不存在被单一厂商垄断的风险

## 4. 模块规划（按层次分类）

### 4.1 通信层 Modules

| ID | 名称 | 开源许可 | 底层实现 | 说明 |
|----|------|---------|---------|------|
| `p2p` | P2P 网络 | AGPL-3.0 | libp2p | 去中心化节点发现与连接 |
| `tor` | TOR 匿名网络 | AGPL-3.0 | arti | 洋葱路由匿名流量 |
| `i2p` | I2P 隧道 | 待定 | 待定 | 匿名网络隧道（规划中） |

### 4.2 加密层 Modules

| ID | 名称 | 开源许可 | 说明 |
|----|------|---------|------|
| `crypto` | 加密管道 | 商业授权 | 端到端加密，可选高级模块 |

### 4.3 同步层 Modules

| ID | 名称 | 开源许可 | 底层实现 | 说明 |
|----|------|---------|---------|------|
| `crdt` | 文档同步 | AGPL-3.0 | yrs | 最终一致性文档协同 |
| `storage` | 存储管理 | AGPL-3.0 | 待定 | 本地持久化存储抽象 |

> **注：** 模块按需安装，不强制依赖。Core 仅声明最小能力集，其余均为可插拔模块。

## 5. 用户交互流程

### 5.1 安装引导流程

```
第一步：安装 UI
         ↓
第二步：UI 检测 Core 状态
         │
         ├── 未安装 → 提示下载 Core（二维码 / 链接）
         │
         └── 已安装 → 继续
         ↓
第三步：UI 引导用户安装模块（模块市场）
         ↓
第四步：配置模块 → 连接节点 → 开始使用
```

### 5.2 模块配置流程

```
用户选择模块
    ↓
Core 获取模块 manifest，透传给 UI
    ↓
UI 根据 manifest 动态渲染配置面板
    ↓
用户填写配置 → Core 保存 → 模块生效
```

manifest 包含：参数定义、默认值、类型约束、是否必填。UI 无需硬编码任何模块的 UI 逻辑。

## 6. 安全设计原则

| 原则 | 实现方式 |
|------|---------|
| 模块权限显式授予 | Core 维护权限白名单，模块只能访问授权范围内的 API |
| UI ↔ Core 本地通信 | WebSocket 走本地回环（127.0.0.1 / localhost），不暴露公网端口 |
| P2P 流量端到端加密 | 默认使用 WireGuard 协议或 Noise 框架加密 |
| 敏感配置不离 Core | 私钥等敏感数据由外部密钥管理器处理，Core 只持有引用，不存明文 |

## 7. 版本与兼容性

- **Core API 版本号**独立演进，每次breaking change递增主版本号
- **模块声明所依赖的 Core API 版本**，manifest 中明确声明 `core_api_version`
- **Core 拒绝加载不兼容的模块**：版本校验失败时模块不被加载，并返回明确错误信息
- **向后兼容**：Core 保持对同一主版本号内所有次版本号的兼容性

```
模块 manifest 字段：
  core_api_version: "1.2"   # 表示该模块依赖 Core API 1.2
```
