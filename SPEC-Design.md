## 1. 总体设计

### 1.1 项目愿景

Cloudfin 是一个**模块化跨平台通信与同步框架**。

类比目标：

- **Redis** —— 纯 API/CLI 控制，程序化驱动
- **无头浏览器** —— 自动化脚本接管一切

Cloudfin 不做封闭生态。**核心目标：让任意多个设备在任意网络环境下安全、低延迟地同步数据。**

UI 与 Core/Modules 严格分离：UI 是独立进程，可以是任意平台的原生应用，也可以是 CLI 工具。

一切围绕这个目标展开：数据怎么存、怎么传、怎么一致、怎么扩展，都服务于这个核心承诺。

### 1.2 五层架构总览

```
┌─────────────────────────────────────────────────────┐
│  第一层：UI 层（平台原生，用户体验优先）                  │
│  Android App / iOS App / Qt / CLI …                  │
└─────────────────────────┬───────────────────────────┘
                          │ WebSocket（单一协议）
┌─────────────────────────▼───────────────────────────┐
│  第二层：Core 层（跨平台，纯逻辑，无 UI）                │
│  连接管理 / 路由 / 模块调度 / 配置                       │
└─────────────────────────┬───────────────────────────┘
                          │ 统一接口（Core API）
┌─────────────────────────▼───────────────────────────┐
│  第三层：通信层 Modules（地基/桥梁，网络通路）              │
│  p2p（点对点）/ tor（洋葱路由）/ i2p（隧道）              │
└─────────────────────────────────────────────────────┘
                          ↑ 三层都只和 Core 通信，
                          │ 模块之间不直接通信
┌─────────────────────────▼───────────────────────────┐
│  第四层：加密层 Modules（装修/安全设施）                    │
│  crypto（端到端加密管道）                                │
└─────────────────────────────────────────────────────┘
                          ↑
┌─────────────────────────▼───────────────────────────┐
│  第五层：同步层 Modules（人工作/车通行，核心价值）           │
│  crdt（文档协同）/ storage（持久化存储）                   │
└─────────────────────────────────────────────────────┘
```

三层 Modules 都**只和 Core 通信**，彼此之间不直接通信。分三层纯粹是**功能分类**：

- **通信层** = 地基 + 桥梁，网络通路本身，提供节点互联能力
- **加密层** = 装修 + 安全设施，在通信基础上加一层保护
- **同步层** = 人在工作 + 车在通行，真正有价值的业务在这一层发生

五层各司其职，任一层换语言、换框架，其他层**无需修改**。

### 1.3 核心功能需求

| 需求 | 说明 |
|------|------|
| 跨平台守护进程 | Android / iOS / Windows / macOS / Linux，后台常驻，开机自启 |
| 模块化插件系统 | .so/.dylib/.dll，可按需安装、加载、卸载，与 Core 解耦 |
| UI与Core分离 | 独立进程，WebSocket 单一连接，可选任意技术栈 |
| P2P组网 | 去中心化、NAT穿透、端到端加密 |
| CRDT文档同步 | 最终一致性、离线可用、冲突自动合并 |

### 1.4 基本技术决策

| 问题 | 答案 |
|------|------|
| 为什么用 Rust | 性能优先 + 内存安全 + 跨平台 FFI |
| 为什么五层分离 | 解耦，各自演进，互不影响 |
| 为什么选择 Open Core | 基础开源（AGPL）+ 高级商业 + 防垄断 |

每个模块的 ZIP 包包含两个文件：

## 3. UI 与 Core 通信设计

### 3.1 为什么用 WebSocket

- **全平台通用：** Android / iOS / Desktop / Web，一个协议打天下
- **双向实时推送：** Server → Client / Client → Server 同时支持，无需轮询
- **HTTP/1.1 兼容：** 天然穿透企业代理，兼容现有网络环境

WebSocket 是 UI 与 Core 之间通信的**最小公分母**。

### 3.2 传输与序列化

| 项目 | 选择 |
|------|------|
| 传输层 | WebSocket（ws://127.0.0.1:19001/ws）|
| 序列化 | JSON |
| 心跳 | 每 30 秒 ping，60 秒无响应判定断开 |
| 重连 | 指数退避（1s → 2s → 4s → max 30s）|
| 请求超时 | 5 秒 |

## 4. Core 设计

### 4.1 定位

Core = **无头守护进程框架**，类比 Redis / cURL / 无头浏览器的设计理念——无 UI，纯 API/CLI，程序化控制。

### 4.2 职责

- 连接管理（UI ↔ Core WebSocket）
- 路由（模块间通信中介）
- 模块调度（加载/卸载/生命周期）
- 配置管理（持久化 + 热更新）

### 4.3 为什么用 Rust

| 理由 | 说明 |
|------|------|
| 性能优先 | 与 C/C++ 同级，适合网络 I/O 密集场景 |
| 内存安全 | 无 GC，无数据竞争，无 use-after-free |
| 跨平台 FFI | 与模块系统天然契合 |
| 现代包管理 | Cargo 一站式依赖与构建 |

## 5. Core 与 Modules 通信设计

### 5.1 为什么模块间不能直连

模块间**禁止直连**，所有通信必须经过 Core 中介。理由：

1. **安全隔离：** 模块权限由 Core 显式授予，无 Core 授权则无法跨模块访问数据
2. **可信中介：** Core 是唯一可信实体，防止恶意模块伪装成合法模块
3. **防止隐性依赖：** 若模块间直连，会形成未声明的耦合，任意模块升级都可能破坏其他模块

### 5.2 调用方式

所有模块间调用通过 Core 的 `modules.call` 接口：

```
模块A → Core.modules.call("模块B", method, params) → 模块B
                      ↓
                 结果返回模块A
```

## 6. Modules 设计

### 6.1 模块层次分类

#### 通信层 Modules（地基/桥梁）

| ID | 名称 | 开源许可 | 底层实现 | 说明 |
|----|------|---------|---------|------|
| `p2p` | P2P 网络 | AGPL-3.0 | libp2p | 去中心化节点发现与连接 |
| `tor` | TOR 匿名网络 | AGPL-3.0 | arti | 洋葱路由匿名流量 |
| `i2p` | I2P 隧道 | 待定 | 待定 | 匿名网络隧道（规划中） |

#### 加密层 Modules（装修/安全设施）

| ID | 名称 | 开源许可 | 说明 |
|----|------|---------|------|
| `crypto` | 加密管道 | 商业授权 | 端到端加密，可选高级模块 |

#### 同步层 Modules（人工作/车通行）

| ID | 名称 | 开源许可 | 底层实现 | 说明 |
|----|------|---------|---------|------|
| `crdt` | 文档同步 | AGPL-3.0 | yrs | 最终一致性文档协同 |
| `storage` | 存储管理 | AGPL-3.0 | 待定 | 本地持久化存储抽象 |

> **注：** 模块按需安装，不强制依赖。Core 仅声明最小能力集，其余均为可插拔模块。

### 6.2 模块分发格式

每个模块 ZIP 包包含：
- **`{模块名}.so`**：二进制插件
- **`{模块名}.so.json`**：manifest（包含 UI 声明）

安装目录：`{Core根目录}/Cloudfin/modules/`

### 6.3 模块配置流程

```
用户选择模块
    ↓
Core 获取模块 manifest，透传给 UI
    ↓
UI 根据 manifest 动态渲染配置面板
    ↓
用户填写配置 → Core 保存 → 模块生效
```

## 7. 其他设计

### 7.1 安装引导流程

```
第一步：安装 UI（用户自行完成）
         ↓
第二步：UI 检测 Core 状态
         │
         ├── 未安装 → UI 显示"一键安装"按钮
         │              ↓
         │         用户点击 → UI 检测 OS + 架构
         │              ↓
         │         自动匹配 GitHub/Gitee 对应版本
         │              ↓
         │         自动下载 + 安装（Linux/macOS 需要 root 时弹窗输入密码，用完即删）
         │              ↓
         └── 已安装 → 继续
         ↓
第三步：UI 引导用户安装模块（模块市场）
         ↓
第四步：配置模块 → 连接节点 → 开始使用
```

### 7.2 安全设计原则

| 原则 | 实现方式 |
|------|---------|
| 模块权限显式授予 | Core 维护权限白名单，模块只能访问授权范围内的 API |
| UI ↔ Core 本地通信 | WebSocket 走本地回环（127.0.0.1 / localhost），不暴露公网端口 |
| P2P 流量端到端加密 | 默认使用 WireGuard 协议或 Noise 框架加密 |
| 敏感配置不离 Core | 私钥等敏感数据由外部密钥管理器处理，Core 只持有引用，不存明文 |

### 7.3 版本与兼容性

- **Core API 版本号**独立演进，每次 breaking change 递增主版本号
- **模块声明所依赖的 Core API 版本**，manifest 中明确声明 `core_api_version`
- **Core 拒绝加载不兼容的模块**：版本校验失败时模块不被加载，并返回明确错误信息
- **向后兼容**：Core 保持对同一主版本号内所有次版本号的兼容性

```
模块 manifest 字段：
  core_api_version: "1.2"   # 表示该模块依赖 Core API 1.2
```

### 7.4 平台与架构标识

#### 系统标识

| 操作系统 | 标识 |
|---------|------|
| Windows | `windows` |
| Linux | `linux` |
| macOS | `macos` |
| Android | `android` |
| iOS | `ios` |

#### 架构标识

| CPU 架构 | Release 标识 | 说明 |
|---------|------------|------|
| x86_64 | `amd64` | 桌面/服务器主流 |
| arm64 / aarch64 | `arm64` | Apple Silicon / 服务器 |
| arm64-v8a（Android） | `arm64-v8a` | 安卓完整架构标识 |

> **注意：** `x86_64` 在 UI 显示和 Release 文件名中统一使用 `amd64`。

#### 模块分发文件名格式

```
CloudFin-Mod-{系统标识}-{架构标识}-{模块名}-{版本}.zip
```

示例：
```
CloudFin-Mod-Linux-amd64-P2P-0.1.0.zip
CloudFin-Mod-Android-arm64-v8a-CRDT-0.1.0.zip
```
